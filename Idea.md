## 1. 提案された手法の構成案（Shim層の構築）

通常、アプリは `/dev/uio0` などを介して `mmap` し、特定のメモリアドレスを叩きます。これを、本物のデバイスファイルではなく、**PythonやC++で書かれたシミュレータにリダイレクトする「Shim（詰め木）」**を作ることになります。

### 構成イメージ
1.  **FW（Linuxアプリ）:** 修正なし、あるいは再コンパイルのみ。
2.  **Wrapperライブラリ:** `mmap` や `ioctl` をフックするスタブ。
3.  **回路ロジック:** Verilogから `[Verilator](verilator.md)` 等でC++モデルに変換したもの。
4.  **通信層:** アプリと回路モデルの間を共有メモリ（Shared Memory）やUnixドメインソケットで接続。

## 2. 実現するための3つの鍵

### ① [Verilator](verilator.md) による C++ 化
Verilogのコードをそのまま動かすのではなく、**[Verilator](verilator.md)** というツールを使って C++ のクラスに変換します。これは非常に高速で、サイクル精度を保ったまま実行バイナリに組み込めます。

### ② `LD_PRELOAD` による関数フック
FW（アプリ）のコードを一切書き換えたくない場合、Linuxの `LD_PRELOAD` という仕組みが使えます。
* アプリが呼ぶ標準の `open()` や `mmap()` を、自作の「偽物の関数」に横取りさせます。
* 特定のアドレス範囲（FPGAのレジストリ領域）へのアクセスだけを、[Verilator](verilator.md)で作ったモデルの変数への書き込みに変換します。

### ③ 通信と同期
アプリがレジストリに「1」を書いた瞬間、[Verilator](verilator.md)側のクロックを1進める必要があります。
* **ZeroMQ** や **Shared Memory** を使うと、アプリとシミュレータを別プロセスで動かしつつ、高速にデータをやり取りできます。

## 1. 実現するための3つのレイヤー

この手法を実現するには、Linuxカーネルの中に「本物のハードウェアと通信するフリをして、実はPython/C++のシミュレータと通信する」ブリッジ層を作ります。

### ① 仮想デバイスの定義（Device Tree Overlay）
Yocto Linux上で、存在しないアドレスにデバイス（例えば I2C スレーブや UIO デバイス）を定義した `.dtbo` ファイルをロードします。
* これにより、`/dev/i2c-2` や `/dev/uio0` といったファイルがカーネルによって自動生成されます。

### ② バックエンド・ドライバの選定
ここが肝です。自作ドライバを書くのは大変なので、既存の仕組みを流用します。
* **UIO (Userspace I/O):** 特定のメモリ領域をユーザー空間に露出させます。このメモリ実体を、シミュレータが書き換える共有メモリにしておきます。
* **i2c-stub / i2c-dev:** I2Cの場合は、カーネルの `i2c-stub` モジュールを使うことで、メモリ上に偽のI2Cチップをエミュレートできます。

### ③ シミュレータとの接続
`/dev` を介したアクセスを、裏側で待ち受けている Python や [Verilator](verilator.md) のモデルに橋渡しします。

## 3. なぜこの手法が「RetroCoreTracer」的なのか？

この構成の最大のメリットは、**「OSのシステムコールレベルでトラッキングができる」**点にあります。

* **インターセプト:** アプリが `ioctl()` で I2C 通信を投げた瞬間、Python 側でその「パケット」をキャプチャできます。
* **リプレイ:** 実機で起きたエラー時のレジストリ読み書き順序を Python で再現（リプレイ）し、仮想デバイスに流し込んでアプリの挙動を確認できます。

## 4. 課題：タイミングの壁

Device Tree でデバイスをでっち上げるのは比較的簡単ですが、**「割り込み（Interrupt）」**の再現が少し厄介です。

* **解決策:** Python シミュレータ側で「計算が終わったから割り込みを上げたい」となった時、QEMU の API や仮想ドライバを介して、Linux カーネルに `IRQ` をインジェクトする必要があります。

## 1. WSL2 上での「疑似ハードウェア」構成案

実機がないWSL2環境で、アプリに「ハードウェアがある」と信じ込ませるための最短ルートは以下の通りです。

### ① I2C / SPI のエミュレーション
WSL2のカーネルには標準で `i2c-dev` や `i2c-stub` が入っていない場合がありますが、**「ユーザ空間 I2C シミュレータ」**を自作してしまえば解決します。
* **手法:** `/dev/i2c-X` を、本物のデバイスファイルではなく **「名前付きパイプ (FIFO)」** や **「UNIXドメインソケット」** で作ります。
* **Pythonの役割:** Python側でソケットを待ち受け、アプリからの I2C Read/Write コマンドを解釈して、可視化ツールの画面に「温度センサに 0xEF を書き込んだ」と表示し、適切な応答を返します。

### ② AXI-Lite / レジストリ (UIO) のエミュレーション
これが一番「FPGA開発」らしい部分です。
* **手法:** **共有メモリ (`/dev/shm`)** を使います。
* **実装:** 1. Python側で共有メモリを確保し、そこを「FPGAのレジストリ領域」と定義します。
    2. アプリ（FW）側は、その共有メモリを `mmap` します。
    3. アプリがメモリに値を書くと、Python側がそれを検知（ポーリングまたはイベント通知）して動作します。

## 2. 開発スピードを爆速にする可視化ツールの活用

単に動くだけでなく、開発スピードを上げるために Python 側で以下の機能を実装するのがポイントです。

* **「いい加減な」シミュレーション:** Verilogを完ぺきに再現するのではなく、「このレジストリに1を書いたら、3秒後にステータスレジストリがReady(1)になる」といった、**「アプリが期待する挙動」だけをPythonの `if` 文で書く**のです。これでHW完成を待たずにFWをガリガリ書けます。
* **異常系インジェクション:** 実機では再現が難しい「I2Cの通信エラー」や「タイムアウト」を、Pythonの画面上のボタン一つで発生させ、FWが正しくエラー処理するかをテストします。

## 3. 推奨するソフトウェアスタック

WSL2で動かすなら、この組み合わせが現実的です。

1.  **C言語アプリ (FW):** WSL2上の `gcc` でビルド。ターゲットアーキテクチャが ARM の場合は、WSL2上の `arm-linux-gnueabihf-gcc` 等でクロスコンパイルし、実行時だけ **`qemu-arm` (User-mode emulation)** を使うと、OS丸ごと動かすより100倍軽いです。
2.  **通信ブリッジ:** アプリが叩く `ioctl` や `mmap` を、共有メモリやソケットへ飛ばすための小さな「ラッパーライブラリ」を用意する。
3.  **シミュレータ (Python):** * **GUI:** `PyQt` や `Streamlit` で、レジストリの値や波形を可視化。
    * **ロジック:** アプリからの要求に応答を返す。
