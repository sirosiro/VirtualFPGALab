# LD_PRELOAD とは

`LD_PRELOAD` は、LinuxなどのUnix系OSにおいて、**プログラムが起動する際に「本来読み込むべきライブラリよりも先に、特定の共有ライブラリ（.soファイル）を強制的にロードさせる」ための環境変数**です。

一言で言えば、**「既存のプログラムの関数を、ソースコードの書き換えや再コンパイルなしで『乗っ取る（フックする）』ための強力な仕組み」**です。

---

## 1. 仕組み：動的リンクの優先順位を操作する

通常、C言語などで書かれたプログラムが `printf()` や `open()` などの関数を呼び出すとき、OSの動的リンカ（`ld-linux.so`）が標準Cライブラリ（`libc.so`）などからその実体を探し出します。

`LD_PRELOAD` を指定すると、動的リンカの探索順序が以下のように変わります：

1.  **`LD_PRELOAD` で指定されたライブラリ**（← ここが最優先！）
2.  プログラム自体が持っているシンボル
3.  標準ライブラリ（`libc.so` など）

もし `LD_PRELOAD` で指定したライブラリの中に、標準関数と同じ名前の関数（例：`open`）が含まれていれば、プログラムは標準ライブラリではなく、**あなたの用意した「偽物（あるいは拡張版）」の関数**を呼び出してしまいます。

---

## 2. 具体的な活用例

この仕組みは、デバッグ、プロファイリング、あるいはシステムの挙動を一時的に変更する際に非常によく使われます。

* **デバッグとプロファイリング:**
    すべてのメモリアロケーション（`malloc` / `free`）をフックして、メモリリークを検出する（Valgrindなどのツールもこの原理を応用しています）。
* **ファイルアクセスの偽装:**
    プログラムが `/etc/passwd` を開こうとしたとき、別のファイルを読み込ませるように `open()` 関数を差し替える。
* **時刻の固定（libfaketime）:**
    `gettimeofday()` などの時刻取得関数を乗っ取り、特定のアプリケーションに対してだけ「現在は2000年1月1日である」と誤認させる。
* **ネットワークの透過的プロキシ化:**
    通常のアプリケーションに手を加えずに、すべてのソケット通信を特定のプロキシ経由に強制する（`tsocks` など）。

---

## 3. 実装のイメージ（関数フック）

例えば、すべてのファイルオープンをログに記録したい場合、以下のような共有ライブラリを作成します。

```c
// my_hook.c
#define _GNU_SOURCE
#include <dlfcn.h>
#include <stdio.h>

// 本物の open 関数を保持するための型定義
typedef int (*orig_open_type)(const char*, int, mode_t);

int open(const char *pathname, int flags, mode_t mode) {
    printf("Hooked: Opening file -> %s\n", pathname);

    // dlsym を使って「本物」の open 関数のアドレスを探す
    orig_open_type orig_open = (orig_open_type)dlsym(RTLD_NEXT, "open");
    
    // 本物を呼び出して結果を返す
    return orig_open(pathname, flags, mode);
}
```

これをコンパイルして実行時に指定します：
`$ LD_PRELOAD=./my_hook.so ls`
すると、`ls` コマンドがファイルを開くたびに、画面にメッセージが表示されるようになります。

---

## 4. 注意点と制限事項

非常に強力なツールですが、いくつか注意すべき点があります。

* **静的リンクされたプログラムには無効:**
    バイナリ内にライブラリが埋め込まれている（Static Link）場合、動的リンカを介さないためフックできません。
* **セキュリティ（setuid ビット）:**
    セキュリティ上の理由から、`setuid` されたバイナリ（`sudo` など）では `LD_PRELOAD` は無視されます。これがないと、誰でも簡単に root 権限の動作を乗っ取れてしまうためです。
* **ABI の一致:**
    フックする関数の引数や戻り値の型（シグネチャ）が本物と完全に一致していないと、プログラムがクラッシュ（セグメンテーション違反）します。

---

## 5. まとめ

`LD_PRELOAD` は、**「バイナリを一切いじらずに、外部から挙動を注入する」**ための魔法のような機能です。

組み込み開発やデバッグの現場では、特定のライブラリのバグを一時的に回避したり、ログ出力を追加したりするのに非常に重宝します。また、本プロジェクトのような仮想化環境において、特定のシステムコールをシミュレーション環境側にバイパスさせるような用途にも応用できる考え方ですね。
